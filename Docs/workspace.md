# 그라운드 룰

1. 브랜치를 기능 및 인원 단위로 나누기
- 네이밍규칙: 접두사/기능_작성자 (예: feat/player-move_PEY)
- 사용하지 않는 오래된 브랜치는 지워 혼동을 방지 합니다
---

2. 작업영역 준수
- 각자 작업중인 영역을 침범하지 않도록 합시다
- 공동사용 가능성이 높은 에셋을 변경하려면 팀원에게 알립니다
---

3. 커밋 메시지 규칙
- 접두사 feat, fix, refactor 등 작성 [참고링크](https://udacity.github.io/git-styleguide/)
- 작은 변경의 경우 제목으로 설명하고 본문작성은 생략해도 괜찮습니다
---

4. 풀 리퀘스트 규칙
- 병합전에 콘솔 로그를 확인하고 크리티컬한 버그가 없는지 테스트 합니다
- 메인에 직접 푸시를 지양하고 브랜치 단위의 병합을 기본으로 합니다
- 주로 병합 담당자가 병합을 하지만 간단한 수정의 경우 개인의 판단으로 병합할 수 있지만 팀원에게 알려야 합니다
---

5. 코드 관리 규칙
- 변수/매개변수: camelCase, 클래스/메서드/구조체/열거형: PascalCase, 상수: UPPER_CASE
- private 의 경우 앞에 _ 언더바를 붙여서 표기한다 (선택)
- 인터페이스: I로 시작, 추상클래스: Base로 끝남
- 네이밍 규칙을 준수하고 코드의 작명을 명확하게 하여 과한 주석을 피하고 가독성을 높입시다
- 주석을 사용하여 핵심코드와 설명이 필요한 부분을 요약하고 설명합니다
---

6. 프리펩 관리 규칙
- 대분류는 폴더트리의 이름으로 잡고 세부는 '속성_이름_특이사항' 에서 최소 2가지 이상으로 정리합니다
- 예: [Prefabs/Characters/Enemy] 폴더의 Orc_Boss, [Prefabs/UI] 폴더의 Player_Hpbar_Blue)
- 많은 곳에 공통으로 적용되어야 하는 프리펩은 원본을 수정하고, 아니면 복사본/바리언트 로 정리합니다
---

7. 씬 관리 규칙
- 같은 씬을 두명의 작업자가 동시에 작업하지 않으며, 그래야 하면 복사하여 작업후 병합합니다
- 병합은 프리펩 및 코드 단위로 진행합니다
---

8. 데이터 정리
- Imports 폴더의 에셋 공유는 Google Drive 시용합니다
- 폴더 구조를 숙지하고 활용합니다 (WIP에 개인작업 폴더 자유롭게 쓰다가 Project폴더에 정리등)
 ---

9. 네임 스페이스 규칙
- 핵심 코드를 제외한 나머지는 기능단위의 네임스페이스를 사용합니다
- 폴더이름에 맞추어 간단한 이름으로 사용하길 권장합니다
--- 

10. 작업 관리 및 피드백
- 작업을 세부 기능단위로 분리하고 작업 테스크에 올려 팀원과 진행상황을 공유합니다
- 작업 문서를 작성하여 깃허브 Docs 폴더에 함깨 커밋하길 권장합니다
---

## 초기 폴더 구조
```
Docs/              문서정리 폴더
Assets/
├─ Imports/         외부 에셋 (gitignore 대상)
├─ Project/          핵심 폴더
│  ├─ Art/              개인 제작 또는 외부 에셋 수정시
│  │  ├─ Audio/
│  │  ├─ Materials/
│  │  ├─ Meshes/
│  │  ├─ Shader/
│  │  ├─ Sprites/
│  │  └─ Texture/
│  ├─ DB/               스크립터블 오브젝트
│  ├─ Prefabs/
│  ├─ Scenes/
│  └─ Scripts/
├─ Settings/         유니티 세팅
│  ├─ Editor/           에디터 전용 스크립트 (빌드제외)
│  ├─ Input/
│  ├─ TextMesh Pro/
│  └─ URP/
└─ WIP/              테스트/정리전 잡다한거 방치해도 무방한 폴더
```
- 개인 작업용 폴더 WIP에 만들길 권장
- 외부 에셋은 gitignore인 Imports에 두지만, 수정하여 공유가 필요하면 Art 폴더 사용


## 설치된 확장 패키지 (링크 첨부)
[UniTask](https://github.com/Cysharp/UniTask)
- async/await 기반 비동기 라이브러리
- 코루틴을 대체하는 유니티 Awaitable의 기원이라고 주장하며 여러 고급 기능 보유
---
[LitMotion](https://github.com/annulusgames/LitMotion)
- struct 기반 트윈 애니메이션 라이브러리 (UniTask, R3, 커스텀 트윈 지원)
- DOTween등 트윈 애니류 성능 [밴치마크](https://github.com/annulusgames/TweenPerformance)에서 1등이라 주장함
